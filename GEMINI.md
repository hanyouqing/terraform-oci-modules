# GEMINI.md - Project Guidelines & Standards

This document defines the standards, conventions, and workflows for the `terraform-oci-modules` project. All changes and new modules must adhere to these guidelines.

## 1. Project Objective

Create a suite of **production-ready**, **secure**, and **cost-aware** Terraform modules for Oracle Cloud Infrastructure (OCI).
- **Production-Ready**: High availability, security hardening, monitoring, and robust error handling.
- **Always Free Compatible**: Every module must have a configuration or example that fits within OCI's [Always Free Tier](https://docs.oracle.com/en-us/iaas/Content/FreeTier/freetier_topic-Always_Free_Resources.htm).
- **Terragrunt Friendly**: Modules should be designed to be consumed easily by Terragrunt (clean inputs/outputs, no provider configurations in modules).

## 2. Module Structure

Each module must follow this directory structure:

```text
module-name/
├── main.tf             # Core resource definitions
├── variables.tf        # Input variable definitions (with descriptions & types)
├── outputs.tf          # Output definitions (with descriptions)
├── versions.tf         # Provider & Terraform version constraints
├── data.tf             # Data sources (optional, can be in main.tf for small modules)
├── README.md           # Documentation (generated by terraform-docs)
└── examples/
    ├── basic/          # Minimal setup, compatible with Always Free Tier
    │   ├── main.tf
    │   ├── variables.tf
    │   ├── outputs.tf
    │   └── README.md
    └── complete/       # Production-grade setup with all features enabled
        ├── main.tf
        ├── variables.tf
        ├── outputs.tf
        └── README.md
```

## 3. Coding Standards

### Naming Conventions
- **Resources/Data Sources**: Use `snake_case`. e.g., `oci_core_vcn.main` (preferred over `this` if specific, but `this` is acceptable for the primary resource).
- **Variables**: `snake_case`. Descriptive names. e.g., `vcn_cidr_blocks` not `cidr`.
- **Outputs**: `snake_case`. consistent with variable names. e.g., `vcn_id`.
- **Files**: `snake_case` or `kebab-case`. `main.tf`, `network-security-groups.tf`.

### Variables
- **Type Constraints**: ALWAYS specify strict types. Avoid `any`. Use `object({...})` for complex inputs.
- **Descriptions**: REQUIRED for every variable.
- **Defaults**:
    - Provide sensible defaults where possible.
    - **Security-sensitive** variables (e.g., passwords, open CIDRs) must NOT have insecure defaults. Force the user to define them or default to a secure state (e.g., empty list for ingress).
- **Validation**: Use `validation {}` blocks for inputs that have specific formats (CIDRs, OCIDs, names with length limits).

### Outputs
- Expose all useful IDs and attributes.
- Include `description` for every output.
- Do not output sensitive values unless marked `sensitive = true`.

### Tagging
- All resources supporting tags must accept `freeform_tags` and `defined_tags` variables.
- Merge user-provided tags with default module tags:
  ```hcl
  freeform_tags = merge(
    {
      "ManagedBy" = "terraform"
      "Module"    = "terraform-oci-modules/<module-name>"
    },
    var.freeform_tags
  )
  ```

## 4. Security Guidelines

- **Least Privilege**: Default security lists/NSGs should be closed or restricted to VCN CIDR. Avoid `0.0.0.0/0` in default variables unless explicitly named `public_...`.
- **Encryption**: Enable encryption in transit and at rest where configurable.
- **Secrets**: NEVER commit secrets. Use variables without defaults or environment variables for sensitive data.
- **Checkov/TFSec**: Modules must pass security scans. Suppress false positives with comments explaining why.

## 5. Documentation

- **README.md**: Must be generated/maintained via `terraform-docs`.
- **Content**:
    - Description of the module.
    - Architecture diagram (text-based Mermaid or reference).
    - Usage examples.
    - Requirements, Providers, Inputs, Outputs (auto-generated).
    - **Always Free Constraints**: Explicitly state if/how the module fits into the Always Free tier.

## 6. Implementation Workflow

1.  **Analyze**: Understand the OCI resource and its constraints.
2.  **Scaffold**: Create the directory structure.
3.  **Implement**: Write `main.tf`, `variables.tf`, etc.
4.  **Examples**: Create `examples/basic` (Free Tier) and `examples/complete` (Prod).
5.  **Test**: `terraform validate`, `terraform fmt`.
6.  **Document**: Run `terraform-docs`.

## 7. Versioning

- **Terraform**: `>= 1.3.0` (or stricter if needed).
- **OCI Provider**: Pin to a recent major version `~> 5.0` or `~> 6.0`.

## 8. Specific Module Instructions

### VCN Module
- Must support dual-stack (IPv4/IPv6).
- strictly separate public/private subnet logic.
- output all subnet IDs for consumption by other modules.

### Compute Module
- **Always Free**: Support `VM.Standard.E2.1.Micro` and `VM.Standard.A1.Flex`.
- Dynamic injection of cloud-init scripts.

### Database Modules
- **Always Free**: Support Autonomous Database Free Tier settings (`is_free_tier = true`).
